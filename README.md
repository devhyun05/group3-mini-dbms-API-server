## 스레드 풀 설계### 구조 개요``` Client ──► Application (accept loop) ──► Task Queue (32) ──► Worker Threads (N=CPU cores) ──► DB Engine (mutex) ```서버는 멀티스레드 구조로 클라이언트 요청을 병렬 수신 및 처리하지만, **DB 엔진 내부는 전역 mutex에 의해 직렬화**되어 동시에 하나의 SQL만 실행된다. ### 스레드 수를 코어 수로 제한한 이유DB 실행 구간이 mutex로 직렬화되어 있기 때문에, 특정 시점에 실제로 SQL을 실행 중인 워커는 **항상 한 명뿐**이다. 나머지 워커는 mutex 해제를 기다리는 상태가 된다. 이 구조에서 워커를 코어 수 이상으로 늘려도: - DB 처리량은 **증가하지 않는다** (직렬화 구간이 병목). - 대기 워커가 늘어난 만큼 스케줄러의 **컨텍스트 스위칭 비용만 증가**한다. - 락 대기열이 길어져 **꼬리 지연(tail latency)도 악화**될 수 있다. 따라서 스레드 풀 크기는 하드웨어 병렬성의 상한인 **CPU 코어 수**로 고정했다. 이는 "DB 락을 잡은 워커 외의 다른 워커들이 네트워크 I/O, 응답 직렬화 등 mutex 바깥 작업을 동시에 수행할 수 있는 범위"와 정확히 일치한다. ### 구성 값| 항목 | 값 | 비고 | | --- | --- | --- | | 워커 스레드 수 | `sysconf(_SC_NPROCESSORS_ONLN)` | 실행 환경의 CPU 코어 수에 자동 맞춤 | | Task Queue 용량 | 32 | Bounded queue — 초과 시 생산자 거절 | | 동기화 | `pthread_mutex_t` + `pthread_cond_t` | 큐 보호 및 빈 큐 대기 처리 |

## 3) 락 방식 정리

| 구간 | 락 종류 | 목적 | 동작 방식 |
|---|---|---|---|
| 작업 큐 접근 | `queue->mutex`  | 큐 데이터 경쟁 방지 | `push/pop` 시 mutex 잠금 |
| 파일/테이블 접근(DB 실행) | `engine->execute_mutex` | 파일/테이블 상태 일관성 유지 | `db_engine_execute()` 전체 구간을 단일 mutex로 보호(사실상 직렬 실행) |





1. **큐 락**으로 멀티스레드 요청 분배를 안전하게 처리
2. **DB 실행 락**으로 파일/테이블 접근을 안전하게 직렬화

즉, **네트워크 처리의 동시성**과 **데이터 일관성** 사이에서 균형을 맞춘 설계
